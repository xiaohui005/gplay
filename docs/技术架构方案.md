# 股票智能研判与买卖决策辅助系统 — 技术架构方案

> 基于《股票智能研判与买卖决策辅助系统实施方案》v1.0  
> 本文档覆盖技术选型、模块设计、数据流、部署结构、测试策略和风险点，不含代码。

---

## 1. 前端技术栈建议

### 1.1 核心框架

| 技术 | 选型建议 | 理由 |
|---|---|---|
| UI 框架 | React 18 + TypeScript | 生态成熟、组件库丰富、适合中后台 + 面向用户混合场景 |
| 构建工具 | Vite 5 | 开发热更新快、构建体积小、原生 ESM 支持 |
| 路由 | React Router v6 | 嵌套路由、loader/data 模式适合个股详情等动态页面 |
| 状态管理 | Zustand | 轻量、TS 友好、按需订阅避免不必要的渲染 |
| 样式方案 | Tailwind CSS 3 + CSS Modules | 快速原型 + 组件级样式隔离；后台可复用 Ant Design |

### 1.2 关键依赖

| 用途 | 推荐库 | 说明 |
|---|---|---|
| 可视化 | ECharts + 自定义 K 线插件 | 行业标准、A 股 K 线、量价图、资金流向图生态成熟 |
| HTTP 请求 | TanStack Query (React Query) | 缓存/去重/重试/乐观更新，适合行情轮询场景 |
| 表格 | @tanstack/react-table | 头等舱级表格控制，适合搜索结果和预警列表 |
| 表单 | React Hook Form + Zod | 预警设置表单校验 |
| 国际化 | 本期不需要，i18n 预留接口 | MVP 只做中文 |
| 组件库 | headless UI（面向用户页）+ Ant Design（后台） | 面向用户页自定义风格强；后台直接复用 Ant Design 提升效率 |

### 1.3 架构要点

- **页面级别按 Lazy Load 分割**：搜索页、个股详情页、研判页、预警中心、后台各模块独立 chunk
- **数据获取模式**：TanStack Query 管理所有 API 调用，个股详情设置 `staleTime: 60_000`（1 分钟缓存），行情轮询用 `refetchInterval`
- **状态归置**：股票搜索历史放 localStorage；用户登录态放 httpOnly cookie + 前端 user context（Zustand）
- **风险提示兜底**：路由守卫 + 组件级条件渲染，任何页面加载时检查 riskLevel 和用户确认状态

## 2. 后端技术栈建议

### 2.1 核心框架

| 技术 | 选型建议 | 理由 |
|---|---|---|
| 开发语言 | Go 1.22+ 或 Java 17+ (Spring Boot 3) | 二者均可。Go 适合高并发采集调度和轻量 API；Java 生态更适合策略规则的复杂配置和审计体系 |
| HTTP 框架 | Go: Gin/Fiber | Java: Spring Boot 3 WebFlux（可选的响应式） |
| ORM | Go: GORM / SQLx | Java: MyBatis-Plus / JPA | 选择与语言一致即可 |
| API 文档 | Swagger / OpenAPI 3.0 | 接口契约与前端/测试对齐 |
| 任务调度 | Go: asynq + Redis | Java: XXL-Job / Quartz | 定时采集、预警检查区别于 Web 请求，需独立调度器 |

### 2.2 中间件与基础设施

| 组件 | 建议 | 用途 |
|---|---|---|
| 缓存 | Redis 7+ | 行情快照缓存（TTL 60s）、数据源限频计数器、分布式锁 |
| 消息队列 | RabbitMQ 或 Redis Stream | 数据采集完成 → 通知研判引擎；预警触发 → 通知推送 |
| 搜索引擎 | 本期不需要，预留 | 股票搜索初期用数据库 LIKE + 拼音映射表即可 |
| 数据源限频 | 令牌桶（Redis 实现或 Go rate limiter） | 保护外部数据源接口不被超频调用 |
| 配置中心 | 后台数据库配置表 + 本地缓存 | 策略配置、文案配置、敏感词从数据库读取并缓存到 Redis |

### 2.3 关键设计决策

- **行情快照读写分离**：写入走采集任务（批量 upsert），读取走 API（缓存穿透时回源数据库）
- **研判计算隔离**：研判引擎作为一个独立模块（非独立服务也可），与 API 层通过函数调用或内部 RPC 通信，方便后续独立部署或灰度
- **审计日志异步写入**：审计日志通过 MQ 异步刷盘，不阻塞主流程
- **敏感词校验门槛**：编译期加载敏感词到内存 trie 树，支持热更新（每分钟检查版本号）

## 3. 数据库选型

### 3.1 推荐方案

| 存储类型 | 建议 | 存储内容 | 理由 |
|---|---|---|---|
| 关系型数据库 | PostgreSQL 16 | stock_basic、strategy_config、sensitive_word、user/role/order、audit_log | 事务支持、JSONB 字段（raw_payload）、丰富的索引类型、无 license 限制 |
| 时间序列数据库 | TimescaleDB（PG 插件）或直接 PG 分区表 | stock_quote_snapshot、stock_kline_daily、stock_capital_flow | MVP 数据量不大，PG 分区表足够；数据量大后平滑迁移 TimescaleDB |
| 缓存 | Redis 7+ | 行情快照、采集限频锁、登录 session、配置缓存 | 毫秒级读写、分布式锁、TTL 自动过期 |
| 文档存储 | 暂不需要 | — | 如果公告/新闻数据量极大可引入 MongoDB，MVP 阶段 PG JSONB 应付 |

### 3.2 选型依据

| 考量 | 结论 |
|---|---|
| 数据规模 MVP 预估 | A 股 ~5300 只股票，每日 1 次 K 线 = ~5300 行；5 分钟延迟行情 ~15 万行/日；PG 完全胜任 |
| 事务需求 | 策略配置保存、采集批次管理需要事务回滚 |
| 查询模式 | 个股详情 = 单行查询 + 少量聚合；采集监控 = 批量过滤查询；PG 索引优化空间大 |
| 团队熟悉度 | PG 是通用选择，社区支持最强 |
| 运维成本 | 单一数据库减小运维复杂度；Redis 作为独立缓存运维成本低 |

## 4. 数据采集任务设计

### 4.1 任务分类与调度策略

| 任务组 | 任务 | 调度模式 | 频率 | 依赖关系 |
|---|---|---|---|---|
| 基础数据 | 股票基础信息同步 | 定时 Cron | 每日 08:00 | 无 |
| 交易日数据 | 交易状态同步 | 定时 | 盘前 08:30 / 盘中 10:00-15:00 每 5min / 盘后 15:30 | 依赖基础信息就绪 |
| 盘中数据 | 延迟行情采集 | 定时 | 1-5 分钟（按授权） | 无 |
| 盘后数据 | 日 K 采集 | 定时 | 每日 16:30+ | 依赖当日行情就绪 |
| 事件数据 | 公告增量采集 | 定时 + 推送 | 10-30 分钟 | 无 |
| 事件数据 | 风险事件采集 | 定时 + 事件触发 | 每日 1 次 + 公告触发 | 依赖公告采集 |
| 财务数据 | 财务数据采集 | 定时 | 每日 1 次（财报期加密） | 无 |

### 4.2 任务执行流程

```
调度器(Cron) 触发 → 读取 data_source_config
  → 鉴权/限频检查 → 调用数据源 API
  → 原始响应写入 raw_market_data（raw 层）
  → 字段映射 + 格式转换（标准化层）
  → 数据质量校验
    → 通过: 写入标准表（upsert by symbol + dataTime + source）
    → 失败: 写入 data_collection_error + 告警通知
  → 生成 batchId + 更新 data_collection_job 状态
  → 发 MQ 消息通知下游（研判引擎/预警服务可消费）
```

### 4.3 幂等与去重

- 同一 `source + symbol + dataType + dataTime` 的重复写入做 upsert
- 采集任务支持 `idempotentKey`（由调度器生成），MQ 消费方做幂等校验
- 补采任务与正常调度共享同一幂等逻辑

### 4.4 监控与告警阈值

| 指标 | 告警阈值 | 通知方式 |
|---|---|---|
| 任务连续失败 | 连续 3 次 FAILED | 钉钉/企微机器人 + 后台告警记录 |
| 数据延迟 | 超过配置阈值（行情 5min / K 线 2h / ...） | 后台延迟看板标红 |
| 补采频率 | 单日手动补采 > 5 次 | 通知数据管理员确认 |
| 采集覆盖率 | 某数据类型成功条数 < 预期 90% | 后台告警记录 |

## 5. 研判引擎模块设计

### 5.1 模块架构

```
┌──────────────────────────────────────────┐
│              研判引擎入口                   │
│  接收: symbol + dataVersion + 策略版本      │
└────────────┬─────────────────────────────┘
             │
    ┌────────┴────────┐
    │  风控前置检查     │  ← 停牌/ST/退市/数据不足 → 直接短路输出
    └────────┬────────┘
             │ 通过
    ┌────────┴────────┐
    │  评分层           │
    │  TrendScorer     │ ← stock_kline_daily
    │  VolumePriceScorer│ ← stock_quote_snapshot
    │  CapitalScorer   │ ← stock_capital_flow (P1)
    │  SectorScorer    │ ← stock_sector
    │  FundamentalScorer│ ← stock_financial
    │  EventScorer     │ ← stock_announcement + stock_risk_event
    │  RiskPenalty      │ ← stock_risk_event + tradeStatus
    └────────┬────────┘
             │ 各因子评分(0-100)
    ┌────────┴────────┐
    │  规则层 (总分+映射) │  ← strategy_config.weights + thresholds
    │  总合公式: Σ(评分×权重) - riskPenalty
    │  建议映射: 总分区间 → suggestion 枚举
    └────────┬────────┘
             │ suggestion + buyPlan/sellPlan
    ┌────────┴────────┐
    │  解释层 (大师模板)  │
    │  7 类 MasterTemplate 按评分/依据填充文案
    └────────┬────────┘
             │ masterGuidance
    ┌────────┴────────┐
    │  聚合输出 + 日志记录 │ → stock_analysis_result
    └─────────────────┘
```

### 5.2 评分器接口抽象

每个评分器实现统一接口：

```go
// 伪代码示意
type Scorer interface {
  Score(ctx, symbol) (ScoreResult, error)
  Type() FactorType
}
type ScoreResult struct {
  Score   float64 `json:"score"`   // 0-100
  Details []Detail `json:"details"` // 每条评分依据
}
```

- 评分器可单独启用/禁用（后台配置）
- 评分器可独立降级：数据不足时返回 score=0 + 标记 disabled
- 新增因子评分器只需实现接口 + 注册

### 5.3 建议映射规则实现

建议映射表（Section 19.3）实现为可配置规则引擎：

```
规则链 (Rule Chain):
  Step 1: 强风控命中 → AVOID / STOP
  Step 2: 数据不足 → WATCH / INSUFFICIENT_DATA
  Step 3: totalScore < 45 → AVOID
  Step 4: 45 <= score < 55 → WATCH
  Step 5: 55 <= score < 65 → BUY_WATCH
  Step 6: 65 <= score < 75 → BUY_LIGHT
  Step 7: score >= 75 + 趋势/量价/资金 ≥2 强 → 回踩/确认关注
  Step 8: 持仓场景 + 跌破止损 → SELL
  Step 9: 高位滞涨/风险升高 → REDUCE
```

- 规则链支持在后台调整阈值（不调整逻辑）
- 每个步骤输出 condition 和失效条件
- 拦截性规则（Step 1-2）不走后续步骤

### 5.4 策略版本控制

```
保存策略 → 生成 strategyVersion (int 递增)
当前策略缓存到 Redis (key: "strategy:active", value: {version, weights, thresholds})
每次研判:
  1. 从请求或缓存读取当前 strategyVersion
  2. 使用该版本策略计算
  3. 研判结果写入 stock_analysis_result (含 strategyVersion)
历史结果不变:
  查询时按 strategyVersion 归档，不做重新计算
```

## 8. 部署结构

### 8.1 MVP 部署拓扑（单机/小型云）

```
┌──────────────────────────────────────────────────────┐
│                     反向代理 (Nginx 1.26+)             │

         │              │

│  Frontend      │  │  Backend API   │
│  Nginx static  │  │  Go/Java App   │
│  dist/         │  │  :8080         │
└───────────────┘  └───┬───────────┘
                       │
          ┌────────────┼────────────┐

```

### 8.2 资源规格建议

| 组件 | MVP 规格 | 理由 |
|---|---|---|
| 应用服务器 | 2C4G × 1 台 | Web 请求 + 采集 + 研判计算，负载不高 |
| 数据库 | 2C4G × 1 台（PG + Redis 同机或分离） | 数据量小，单机可负担 |
| 磁盘 | 100GB SSD | 日志 + raw 层数据主要占空间，定期归档清理 |
| 带宽 | 5Mbps | 行情数据按 KB 级传输，不高 |

### 8.3 环境规划

| 环境 | 用途 | 数据源策略 |
|---|---|---|
| dev | 开发联调 | AKShare / Tushare 测试账号 / mock |
| test | QA 测试 | Tushare Pro / 同类低成本源 |
| staging | 预发布验证 | 同生产（或延迟行情） |
| production | 正式上线 | 授权行情 + 商业数据商 |

### 8.4 关键部署决策

- **前端静态资源**：Nginx 托管，CDN 可选
- **后端无状态**：API 服务水平扩展，session 存 Redis
- **采集任务实例锁**：Redis 分布式锁确保同一采集任务同时只有一个实例执行
- **日志集中化**：应用日志 stdout → 文件，Elastic Stack（P1）
- **数据库备份**：PG 每日自动 pg_dump + WAL 归档

## 9. 测试策略

### 9.1 分层测试

| 层级 | 工具/框架 | 覆盖范围 | MVP 目标覆盖率 |
|---|---|---|---|
| 单元测试 | Go: `testing` + testify / Java: JUnit 5 + Mockito | 评分器、规则映射、风控拦截、数据清洗函数 | 核心逻辑 >= 80% |
| 接口测试 | Postman / Newman 或 Karate | 全部 API 的正例、异常、降级场景 | 100% 覆盖接口清单 |
| 集成测试 | Testcontainers（PG + Redis）或 docker-compose | 采集流程：API → raw → 清洗 → 入库；研判：评分 → 建议 → 输出 | 核心流程每条至少 1 个 |
| 端到端测试 | Playwright | 搜索 → 个股详情 → 研判 → 预警设置；管理员采集监控 | 6 条核心用户路径 |
| 数据质量测试 | 自定义断言 + CI 定时 | 每日采集结果：字段完整率、延迟分布、异常占比 | 自动报表 |

### 9.2 核心测试场景与验收标准对照

| 测试用例 ID | 文档用例 | 层级 | 关键验证点 |
|---|---|---|---|
| T001 | T001 - 搜索有效 A 股 | 接口 | 代码/名称/拼音返回同一结果 |
| T002 | T002 - 正常交易股票 | E2E | 大师指导 + 评分 + 条件清单 + 建议动作 |

### 9.3 异常测试矩阵

| 异常场景 | 预期行为 | 测试方式 |
|---|---|---|
| 数据源 API 超时 | 采集任务标记 FAILED，按策略重试 | mock 接口延迟 > 超时配置 |
| 数据源返回空数据 | 任务标记 PARTIAL_SUCCESS，记录明细 | mock 返回空数组 |
| Redis 不可用 | 缓存穿透回源 DB + 降级日志 | 关闭 Redis 实例 |
| PG 连接池耗尽 | 返回 503 + 错误码 3001 | 用尽连接池 |
| 并发补采同一股票 | 幂等去重，不产生重复数据 | 同时发送相同补采请求 |
| 行情数据部分缺失（如 4000/5300 成功） | 成功部分入库，失败部分告警 | mock 部分股票失败 |

### 9.4 自动化测试流水线

```
Push → lint + unit test → build → integration test (testcontainers)
  → API contract test (Newman) → security scan (SAST)
  → deploy to test → smoke test (Playwright)
  → deploy to staging → E2E test → release gate
```

## 10. 风险点

### 10.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|---|---|---|---|
| 免费数据源不稳定/限频/接口变更 | 采集失败，研判数据缺失 | 高 | 多数据源冗余 + 延迟重试 + 前台降级展示；正式环境切换授权数据源 |
| 评分规则初期不准 | 建议质量低，用户不信任 | 中 | 策略版本灰度 + 规则兜底 + 条件化表达不承诺准确；上线后持续调优 |
| 实时行情延迟 | 数据不实时，用户不满 | 中 | 前台明确展示 dataTime + delayMinutes；降低建议强度；P1 再评估实时需求 |
| 数据库性能瓶颈 | 行情写入频繁影响查询 | 低（MVP） | Redis 缓存读链路；写入批量 upsert；读写分离 P1 |
| 采集任务竞争 | 同一任务多副本执行 | 低 | Redis 分布式锁 + idempotentKey |

### 10.2 业务合规风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 用户误读为确定性投资建议 | 合规/投诉风险 | 条件话表达 + 免责声明 + 风险弹窗 + 敏感词拦截 + 日志留痕 |
| 数据源授权边界不清 | 数据来源不合法 | 上线前确认数据源白名单和授权合同；免费接口只用于开发和内测 |
| 敏感词漏拦截 | 违规文案发出 | 敏感词库持续更新 + 发布前校验 + 事后审计检查 |
| 付费/免费权益越权 | 付费墙失效 | 后端统一权限校验中间件 + 前端路由守卫双层保险 |

### 10.3 项目管理风险

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 数据源采购周期超 6 周 | MVP 上线阻塞 | 第 1 周同步启动商务询价 + 测试账号申请；测试环境先以低成本源验证 |
| 数据字段与文档不一致 | 接口联调返工 | 第 1 周出完整数据字段映射表，前后端/数据按映射表对齐 |
| 评分规则量化困难 | 建议质量不达标 | 默认权重可配置；MVP 以"趋势+量价+基本面"为主，资金和情绪事件降级 |
| 团队协同效率 | 前后端/数据配合问题 | 接口契约优先（OpenAPI 3.0）；数据字段映射表共享；
